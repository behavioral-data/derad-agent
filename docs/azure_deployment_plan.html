00 · Target shape

Architecture

One Linux container running gunicorn + Flask on Azure App Service. The container bakes the 712 MB notes index, so cold start is single-step. The webhook returns 200 immediately and the ~10 s pipeline runs in a background thread inside the same worker. Idempotency, rate-limits, and the engagement schedule live in Azure Table Storage. Secrets in App Service application settings (Key Vault later).

Compute

Azure App Service Linux, custom container, plan B2 / B3 (4–8 GB RAM). Single instance. gunicorn 1 worker × N threads — index is in-memory, replicating it across workers would multiply RAM 4×.

Storage

Index baked into the image (indexes/notes_index/). Azure Table Storage for dedup, rate-limits, mention log, engagement schedule. No blob mount, no Azure Files, no Cosmos.

Async

threading.Thread kicked off after the 200 returns. Acceptable to lose one in-flight reply on redeploy (research scale). Skip Service Bus / queues until volume forces it.

Secrets

Azure OpenAI key + endpoint, X consumer key/secret, three pairs of OAuth1 access token/secret (one per bot). App Service application settings for v1; Key Vault references are a one-line upgrade.

Observability

Application Insights for traces + exceptions, Log Analytics for App Service logs, Azure cost alerts (OpenAI usage cap), simple counters in Table Storage for mentions-per-day per persona.

Identity

Three X developer-managed bot accounts. Same X App registration. One Account Activity dev environment label. Each bot user subscribed once via API.

Phase 0 · Gates

Prerequisites — verify before writing infra

These are kill switches, not nice-to-haves. Each one can block or fundamentally change the architecture.

The current code uses tweet_create_events, which is the Account Activity API. Historically only Enterprise tier; the current Pro tier offers a narrower variant. 3.4 · Deployment workflow · runs on your Azure-auth'd machine

Need az, azd, and Docker on the host that runs this. The index dir (indexes/notes_index/) must be present locally — Docker bakes it in.

Order matters. azd provision creates the App Service with Key Vault references to secrets that don't yet exist. Until those secrets are seeded, the App Service starts in a faulted state (every reference resolves to the literal @Microsoft.KeyVault(...) string). The runbook below seeds secrets before azd deploy, then restarts the App Service so it re-resolves references with the now-existing secrets before the production image takes over. 
# 0. Tools (one-time per machine)
curl -fsSL https://aka.ms/install-azd.sh | bash      # azd
# az: see https://learn.microsoft.com/cli/azure/install

# 1. Auth + env
az login
azd auth login
azd env new derad-prod
azd env set AZURE_LOCATION eastus     # match the existing Azure OpenAI deployment

# 2. Preview infra what-if (read-only)
azd provision --preview

# 3. Provision (creates RG, identity, ACR, Storage, KV, Log Analytics,
#    App Insights, App Service Plan, App Service). App Service is faulted
#    until step 4 completes.
azd provision

# 4. Seed Key Vault secrets — must complete BEFORE azd deploy.
KV=$(azd env get-values | grep AZURE_KEY_VAULT_NAME | cut -d= -f2 | tr -d '"')
for kv_name in azure-openai-api-key azure-openai-endpoint azure-openai-deployment-embed azure-openai-deployment-chat \
               x-api-key x-api-secret \
               x-access-token-agreeable x-access-token-secret-agreeable \
               x-access-token-neutral   x-access-token-secret-neutral   \
               x-access-token-satirical x-access-token-secret-satirical \
               bot-user-id-agreeable bot-user-id-neutral bot-user-id-satirical; do
  read -rsp "value for $kv_name: " VAL; echo
  az keyvault secret set --vault-name "$KV" --name "$kv_name" --value "$VAL" >/dev/null
done

# 5. Verify all 15 secrets seeded.
test $(az keyvault secret list --vault-name "$KV" --query "length(@)" -o tsv) -eq 15 \
  && echo "✓ 15 secrets present" || echo "MISSING secrets — fix before deploy"

# 6. Restart App Service so it re-resolves Key Vault references with the
#    secrets in place. Without this, the first /healthz call may still see
#    cached unresolved values.
APP=$(azd env get-values | grep AZURE_APP_SERVICE_NAME | cut -d= -f2 | tr -d '"')
RG=$(azd env get-values | grep AZURE_RESOURCE_GROUP   | cut -d= -f2 | tr -d '"')
az webapp restart --name "$APP" --resource-group "$RG"

# 7. Build image + push to ACR + update linuxFxVersion + restart.
#    Until this completes, the App Service is configured to pull
#    <acr>/derad-agent:latest but no such tag exists yet, so the container
#    fails to start. That's expected and is resolved by this step.
azd deploy

# 8. Verify
HOST=$(azd env get-values | grep AZURE_APP_SERVICE_HOSTNAME | cut -d= -f2 | tr -d '"')
curl -s "https://$HOST/healthz"           # → {"ok":true,"index_loaded":false}
sleep 30
curl -s "https://$HOST/healthz"           # → {"ok":true,"index_loaded":true}
curl -s "https://$HOST/info?reply_id=R1&tone=neutral" | head -c 400

4.2 · Capture bot user IDs & seed them into Key Vault

The self-reply guard fails closed without BOT_USER_ID_<TONE> set. Use the CLI's me subcommand to retrieve each bot's id from its OAuth1 creds, then write it to Key Vault as the corresponding secret. App Service picks them up on restart.

for TONE in agreeable neutral satirical; do
  ID=$(derad-webhooks me --tone "$TONE" | python -c "import json,sys; print(json.load(sys.stdin)['id'])")
  echo "  $TONE -> $ID"
  az keyvault secret set --vault-name "$KV" --name "bot-user-id-$TONE" --value "$ID" >/dev/null
done
az webapp restart --name "$APP" --resource-group "$RG"
4.3 · Register the three webhook URLs

X immediately issues a CRC GET on each registration call. The App Service must already respond with a valid HMAC (4.1 confirms this). One webhook per tone, each using that tone's OAuth1 credentials.

for TONE in agreeable neutral satirical; do
  derad-webhooks register --tone "$TONE" --url "$HOST/mention-$TONE"
done

# Capture the assigned webhook IDs for the next step.
derad-webhooks list --tone neutral     # all three tones share the same X dev environment
4.4 · Subscribe each bot user

Registration alone doesn't deliver events. Each bot user has to subscribe to its webhook before its mentions are routed. The subscribe call uses the bot's own OAuth1 tokens, so --tone selects the principal.

AGREEABLE_WEBHOOK_ID=... NEUTRAL_WEBHOOK_ID=... SATIRICAL_WEBHOOK_ID=...

derad-webhooks subscribe --tone agreeable --webhook-id "$AGREEABLE_WEBHOOK_ID"
derad-webhooks subscribe --tone neutral   --webhook-id "$NEUTRAL_WEBHOOK_ID"
derad-webhooks subscribe --tone satirical --webhook-id "$SATIRICAL_WEBHOOK_ID"

# Sanity: each webhook should have exactly one subscription (the bot itself).
for TONE in agreeable neutral satirical; do
  derad-webhooks subscriptions --tone "$TONE" --webhook-id "$(eval echo \$${TONE^^}_WEBHOOK_ID)"
done

Phase 4.5 · Research data · next

Append-only event log for the study

What we persist today is enough to run the bot, not enough to study it. The dedup table records "we saw this mention_id" and the rate-limit table records "this author hit us"; everything else useful — parent text, cited notes, reply ids, timings, drop reasons — only exists as log lines in Application Insights, which is queryable but messy and 30-day-retained. This phase adds a real structured event log before Phase 5 cutover so we capture data from the first mention forward.

Design decision. One more Azure Tables table (no new resource type, no DB to operate). Tables are free at our scale, atomic on insert, and pull cleanly via the SDK or az storage entity query. If the data ever outgrows Tables, exporting to Parquet is one script away.

Where the money actually goes

Variable costs that will dominate the bill once traffic exists.

Azure OpenAI tokens

3 LLM calls per mention: planner (short), relevance filter (medium), composer (longest). Rough order: $30–150/month at 100 mentions/day across the three bots — depends heavily on which deployments are used. The composer accounts for ~70% of the spend.

Azure OpenAI embeddings

One embed call per planner query × ~3 queries per mention. Embedding tokens are ~10× cheaper than chat tokens; this is essentially free at our scale (<$5/month).

Storage + telemetry

Standard LRS storage with the four tables we need: cents per month. Application Insights basic ingestion (first 5 GB free): also negligible.

Cost levers (apply if/when the bill matters)

Listed in order of effort vs. payoff.

  • cheap UW research credits / Azure for Students. Likely $100–200 free; covers the App Service Plan outright. Check before paying for anything.
  • cheap Two-model split. Planner + relevance filter run on a cheaper chat deployment (e.g. gpt-5-mini); composer stays on the strong model. Estimated 40–60% reduction on token spend.
  • medium Per-claim reply cache. Hash the parent-post text; identical claims skip the entire pipeline and reuse the previous reply. Huge saver when a viral claim gets repeatedly fact-checked. Already in the post-v1 backlog.

Design decisions & explicit non-choices

What we considered and rejected — so we don't re-litigate later.

App Service over Container Apps

Container Apps adds scale-to-zero and revisions; we need neither (steady, low traffic; one bot fleet). App Service Linux with a custom container is simpler and cheaper at this scale. Revisit if traffic warrants horizontal scaling.

Bake index in image, not Azure Files

Mount paths add cold-start time and a failure mode (mount not ready when worker imports the module). Community Notes data updates ~monthly; rebuilding the image is acceptable. Image stays well under 5 GB.

threading.Thread over Service Bus

Async durability matters when losing a job is unacceptable. For a research bot, a missed reply during a redeploy is fine. A queue + worker doubles the moving parts and the cost for no near-term gain. Revisit if we ever need cross-instance fan-out.

Table Storage over Cosmos / Postgres

Everything we need to persist is keyed lookups and append-only logs. Table Storage costs cents per month, has 1 ms reads, and the SDK is one import. Cosmos and Postgres pay for capabilities we don't use.

App Service settings over Key Vault (for v1)

Key Vault references are a one-line upgrade per setting and we'd rather not gate launch on the IAM round-trip. Revisit before extending access to non-research collaborators.

Single gunicorn worker

The 1.2 GB embedding matrix is per-process. Two workers ≈ 2.4 GB just for the index. -w 1 --threads 8 keeps RAM down and lets the GIL release during the LLM/HTTP waits, which is where the time goes anyway.
