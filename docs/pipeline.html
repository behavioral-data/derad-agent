Fact-check pipeline

An invoked-bot architecture for fact-checking X posts. Nine stages move a mention from the X filtered stream through web-grounded verification, multimodal extraction, and tone-controlled rendering to a posted reply.

03 · zoom — stage 4

Inside the iterative verification loop

Stage 4 isn't a single search. A Claude controller decides — one step at a time — whether to keep searching, what to search for next, or whether the evidence is sufficient.

Caps

  • 5 max questions per claim
  • 3 hits per question (top-k)
  • 60 s wall-clock budget
  • 120 s per-call LLM timeout

Stop signals

  • stop ≥2 fact-checker or reputable-news hits with consistent stance
  • stop multiple searches, no relevant coverage → declare NEI
  • stop wall-clock or question cap hit

Continue signals

  • go primary source not yet checked (e.g. official press release)
  • go originating publisher unknown — look for retraction
  • go date/context detail would settle ambiguity
  • go author handle's authenticity not yet checked

04 · URLs — trust & persistence

Where URLs come from and how we keep them honest

Where Bing's URLs become evidence, where the small model can fabricate, and what's read vs. saved at every step.

Every cited URL is now fetched and title-matched before it enters evidence. The validator dropped the recent India Today miscite (a fabricated Neuralink slug whose ID actually routed to a 2023 movie article) with a Jaccard score of 0.00 in offline reproduction.

URL lifecycle per Foundry search call

step 1
Query out. Sent to the single-purpose Foundry agent (model gpt-4.1-mini, only the bing_grounding tool attached).
step 2
Bing search. Tool fires; results return to the model internally. Foundry never exposes the raw Bing output to API callers.
step 3
Agent re-emits as JSON. gpt-4.1-mini transcribes URLs into a structured response. Hallucination point. The small model pattern-matches URL shapes and can fabricate plausible slugs / IDs / dates that don't map to real articles.
step 4
Annotations check (empirically empty). Foundry's url_citation annotations would normally be authoritative, but they don't populate for structured-JSON output. So we can't filter on them.
step 5
GET + title extract. Each agent-claimed URL is fetched (≤32 KB body, 6 s timeout, redirects followed, browser User-Agent). The page's <title> is parsed out.
step 6
Title Jaccard match. Tokenize claimed and actual titles (lowercase alphanumeric, ≥3 chars, stopwords + framing words filtered). Overlap ≥ 0.20 → accepted. Below → rejected as title_mismatch with score logged.
step 7
Status check. HTTP 4xx/5xx → rejected. No <title> at all (typical for JS-rendered pages) → passed through unverified and logged.

What we read

  • yes HTTP GET of each cited URL, first 32 KB only — for title extraction.
  • yes Page <title> compared against the agent's claimed title.
  • no Article body / paragraphs (no snippet extraction from real content yet).
  • no Playwright rendering for JS-heavy sites — they fail validation or pass through unverified.
  • no Archive-Today / Wayback cross-checks.

What we don't save

  • no Raw Bing search results (Foundry hides them).
  • no Foundry response objects (only parsed + validated hits).
  • no Fetched HTML / titles used during validation (rejection reasons are logged, not stored).
  • no Per-stage Claude conversation transcripts.
Pragmatic next step. When the agent's claimed snippet doesn't appear (case-insensitive substring) in the validated page's first 4 KB of text, drop the citation as snippet_mismatch. ~30 minutes of code, ~100 ms per URL (we already fetched the body). Catches the case where the URL is correct but the agent paraphrased what's there inaccurately.

05 · stages in detail

What each stage does, with what

One card per stage. Service/model is the actual deployment we hit at runtime.

Stage 1 — Ingestion

data plumbing

The streamer matches @eddiexbot mentions on the X filtered stream; the bot fetches the parent tweet with full author + media + URL context.

  • service X API v2 — filtered stream + posts.get_by_id
  • out TweetSnapshot + tweet_context (handle, bio, verified, created_at, followers, posted_at, expanded URLs, referenced tweets, photos)

~2 s · capped by X read-rate-limit

Stage 1.5 — Multimodal

Claude + Foundry+Bing

Per-image extraction; only fires when the parent carries ≥1 photo.

  • tier 1 OCR — Claude VLM transcribes every visible string
  • tier 2 Description — names public figures, landmarks, events, brands with reasonable confidence
  • tier 3 Provenance approx — description fed back through Foundry+Bing
  • tier 4 Manipulation / AI-gen detection — forced NEI per spec

~7–13 s per image · 5 MB base64 cap (Pillow-resized)

Stage 4 — Iterative verify

Claude + Foundry+Bing

Papelo-style loop: a Claude controller decides continue/stop, issues the next question, Foundry+Bing answers. Hallucinated URLs are filtered by content-based validation (see URLs: trust & persistence).

  • controller Claude Sonnet 4.6, low reasoning
  • search Foundry agent + bing_grounding (gpt-4.1-mini)
  • caps ≤5 questions · ≤3 hits/q · 60 s · 120 s/LLM call

~20–50 s · 1–3 questions typical · early-stops when ≥2 reliable hits agree

Stage 4.5 — Reconcile

Claude reasoning

One Claude call folds Lens 1 (text-text), Lens 2 (image-text), and Lens 3 (cross-modal) over the full evidence + tweet_context bundle. Emits atomic propositions, presentation payload, and per-evidence stances.

  • model Claude Sonnet 4.6, extended thinking (budget=4096)
  • author signal handle + bio + verified + account-age → parody / fan / aggregator detection

~30–50 s · heaviest single LLM call in the pipeline

Stage 4.5.2 — Source reliability

structural rule

Four-step lookup classifies every URL into a tier; structural verdict rule reads from those tiers.

  • 1 Curated IFCN signatory set
  • 2 Wikipedia perennial-sources table
  • 3 Subdomain → parent fallback
  • 4 Batched Claude model-prior, cached per worker
  • verdict Refuted / Supported needs ≥2 fact-checker / reputable-news on the central claim

243 curated domains across five tiers

Stage 5 — Audit

mechanical

Pure-Python audit re-derives the verdict and validates every cited URL is in the source-quality table. CoVe-style Claude audit is a planned follow-up.

  • checks verdict consistency · URL containment · one-bucket rule
  • on fail forces NotEnoughEvidence; NEI-shaped payload swapped in

< 50 ms

Stage 6 — Freeze

data plumbing

Serializes the complete FrozenVerdict Pydantic tree to disk + an event row in Tables. Deep-immutable: nested models are frozen=True and every collection is a tuple.

  • disk data/freezes/<invocation_id>.json
  • tables MentionEvents row
  • pins Claude version · VLM version · source-list snapshot dates · git SHA

~100 ms

Stage 7 — Render

Claude reasoning

Claude renders one reply in the requested tone. The renderer sees ONLY RendererView (presentation_payload + tone_neutral_justification + state) — reasoning fields are unreachable by contract.

  • tones agreeable · neutral · agonistic — three independent prompts, no cross-references
  • invariance verbatim proper nouns · ≥1 cited URL (in verified state) · ≤256 X-weighted chars
  • refusal retry once with harm-reduction nudge; else raise → pipeline_error

~5–10 s · 1–2 retries on overflow

Stage 8 — X poster

data plumbing

Posts the rendered reply as a reply-to the original mention. The bot's /info URL is appended after the rendered text.

  • service X API v2 posts.create · OAuth 1.0a
  • cap 100 tweets / day (X Basic) · 500 soft per-tone (DERAD_MAX_MENTIONS_PER_DAY)
  • dry run DERAD_DRY_RUN=true logs but doesn't post

~1–2 s

Stage 9 — Outcome

data plumbing

Each accepted mention writes one event row. Engagement on each bot reply is polled hourly by a Container Apps cron job.

  • outcomes replied · empty_reply · pipeline_error · parent_fetch_failed · dry_run · pipeline_queue_timeout
  • variables likes · RTs · replies · QTs · target-user downstream sharing-quality (Mosleh)
  • tables MentionEvents · MentionDrops · EngagementSnapshots · BotReplyReplies

Per-mention cost (est.)

  • Foundry+Bing ~3–5 searches × ~$0.025 = ~$0.10
  • Claude 5–10 calls, ~50K tokens total ≈ ~$0.04
  • X API ~$0.013 per posted tweet at $200/100 daily cap
  • ≈ $0.15 / mention

Burst limits

  • 25 concurrent fact-check pipelines (semaphore cap)
  • 10 min queue timeout before drop (pipeline_queue_timeout)
  • 100/day X post cap (hard)
  • 500/day internal soft cap per tone

08 · failure modes

How each stage degrades

Every stage is designed to fail toward NEI rather than toward a confident wrong reply.

Failure Detected by Behavior
Parent fetch returns nothing (deleted / private) Stage 1 Outcome parent_fetch_failed; bot drops the mention silently
Image too large or unreadable Stage 1.5 Image skipped, pipeline continues text-only
VLM extraction error Stage 1.5 This image's record marked unreadable, others continue
Foundry 429 / Bing error Stage 4 / 1.5 That question returns []; iterative controller decides whether to retry or stop
Claude 429 / timeout (120 s) Any LLM stage Call returns error; pipeline records pipeline_error and skips reply
Verdict label disagrees with structural rule Stage 5 audit Verdict forced to NotEnoughEvidence; NEI payload swapped in
Renderer cites a URL not in source-quality table Stage 7 runtime check One retry with the error as feedback; on second failure, empty_reply outcome
Renderer exceeds 280 X-weighted chars Stage 7 runtime check Same as above
Pipeline semaphore queue full for 10 minutes App-level throttle Outcome pipeline_queue_timeout; visibility in events table
X stream disconnect / 429 on reconnect Streamer thread Exponential backoff (capped at 30 min) until reconnect; mentions during disconnect are lost
Worker restart (deploy, scale, crash) App Service In-flight pipelines killed; atexit hook closes X stream so the next worker reconnects fast
