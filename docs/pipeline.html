Fact-check pipeline

An invoked-bot architecture for fact-checking X posts. Nine stages move a mention from the X filtered stream through web-grounded verification, multimodal extraction, and tone-controlled rendering to a posted reply.

03 · zoom — stage 4

Inside the iterative verification loop

Stage 4 isn't a single search. A Claude controller decides — one step at a time — whether to keep searching, what to search for next, or whether the evidence is sufficient.

Caps

  • 5 max questions per claim
  • 3 hits per question (top-k)
  • 60 s wall-clock budget
  • 30 s controller call (was 120 s)
  • seed step 0 = claim text → search (no controller call)

Stop signals

  • stop ≥2 fact-checker or reputable-news hits with consistent stance
  • stop multiple searches, no relevant coverage → declare NEI
  • stop wall-clock or question cap hit

Continue signals

  • go primary source not yet checked (e.g. official press release)
  • go originating publisher unknown — look for retraction
  • go date/context detail would settle ambiguity
  • go author handle's authenticity not yet checked

04 · URLs — trust & persistence

Where URLs come from and how we keep them honest

Where Bing's URLs become evidence, where the small model can fabricate, and what's read vs. saved at every step.

Search backend is now gpt-5.4-mini + the native web_search tool via Azure OpenAI Responses API. URLs arrive via two paths, validated differently. Annotation URLs (server-stamped by the tool runtime) skip HEAD validation by default; markdown-fallback URLs always go through the full title-match validator.

URL lifecycle per search call

step 1
Query out. Sent to responses.create(model=gpt-5.4-mini, tools=[{type: web_search}]). Per-call timeout 90 s.
step 2
Web search. Tool runtime issues the search and feeds results back to the model along with a system-instructed fact-check framing (avoids gpt-5.4-mini refusing on misinformation queries).
step 3a
Annotation path (preferred). Model emits a bullet-list reply; the tool runtime attaches url_citation annotations to each bullet. URL + title come from the runtime, not the model — fabrication is impossible. Marked is_annotation=True.
step 3b
Markdown-fallback path. Some runs emit zero annotations and the model transcribes citations inline as [title](https://…). URLs are model-typed → still possibly correct but subject to fabrication. Marked is_annotation=False.
step 4
Validation in parallel. ThreadPoolExecutor(max_workers=8) fans out per-URL classification: annotation hits short-circuit (no fetch); markdown-fallback hits get GET + first 32 KB + <title> extract + Jaccard match (threshold 0.20). Worst-case validation latency dropped from O(top_k × 6 s) serial to O(6 s) parallel.
step 5
Status & title match. HTTP 4xx/5xx → rejected. Token-overlap below 0.20 → rejected as title_mismatch. No <title> (JS-rendered) → passed through unverified.
override
Paranoid mode. DERAD_VALIDATE_ANNOTATIONS=1 forces HEAD-validation even on annotation-stamped URLs — useful when investigating whether the tool runtime is emitting bad citations.

What we read

  • yes url_citation annotations on every Responses API call — server-stamped URL + title.
  • yes HTTP GET (first 32 KB) of every markdown-fallback URL — for title extraction.
  • yes Page <title> compared against the model's claimed title via Jaccard.
  • no Article body / paragraphs (no snippet extraction from real content yet).
  • no Playwright rendering for JS-heavy sites — they fail validation or pass through unverified.
  • no Archive-Today / Wayback cross-checks.

What we don't save

  • no Raw web-search results (only parsed + validated hits land in evidence).
  • no Responses API response objects (only the SearchHit projection).
  • no Fetched HTML / titles used during validation (rejection reasons are logged, not stored).
  • no Per-stage Claude conversation transcripts.

05 · stages in detail

What each stage does, with what

One card per stage. Service/model is the actual deployment we hit at runtime.

Stage 1 — Ingestion

data plumbing

The streamer matches @eddiexbot mentions on the X filtered stream; the bot fetches the parent tweet with full author + media + URL context.

  • service X API v2 — filtered stream + posts.get_by_id
  • out TweetSnapshot + tweet_context (handle, bio, verified, created_at, followers, posted_at, expanded URLs, referenced tweets, photos)

~2 s · capped by X read-rate-limit

Stage 1.5 — Multimodal

Claude + web_search

Per-image extraction; only fires when the parent carries ≥1 photo. Multi-image tweets run images concurrently via ThreadPoolExecutor(max_workers=4).

  • tier 1 OCR — Claude VLM transcribes every visible string
  • tier 2 Description — names public figures, landmarks, events, brands with reasonable confidence
  • tier 3 Provenance approx — description fed back through web_search
  • tier 4 Manipulation / AI-gen detection — forced NEI per spec
  • timeout 60 s VLM call · 90 s provenance search

~7–13 s per image (concurrent across images) · 5 MB base64 cap (Pillow-resized)

Stage 4 — Iterative verify

Claude + web_search

Papelo-style loop with a Claude controller. Step 0 is a canned seed search on the claim text directly (no controller call — saves one Claude reasoning round-trip per mention). Steps 1..N let the controller condition on history and issue the next question.

  • controller Claude Sonnet 4.6, reasoning_effort=low, timeout 30 s
  • search web_search via Responses API (gpt-5.4-mini)
  • caps ≤5 questions · ≤3 hits/q · 60 s wall-clock

~15–40 s · 1–3 questions typical · early-stops when ≥2 reliable hits agree

Stage 4.5 — Reconcile

Claude reasoning

One Claude call folds Lens 1 (text-text), Lens 2 (image-text), and Lens 3 (cross-modal) over the full evidence + tweet_context bundle. Emits atomic propositions, presentation payload, and per-evidence stances.

  • model Claude Sonnet 4.6, extended thinking (budget=4096), timeout 90 s
  • author signal handle + bio + verified + account-age → parody / fan / aggregator detection
  • skip Structural NEI shortcut: when text_evidence is empty AND no image-provenance URL is in a credible tier, reconcile is bypassed (saves ~5–10 s)
  • prompt diet evidence snippets truncated to 240 chars, source-quality rationales to 120 chars
  • drift stance-count mismatches emit a reconcile_stance_drift metric for observability

~20–40 s · heaviest single LLM call when it fires

Stage 4.5.2 — Source reliability

structural rule

Three-layer cache feeds a structural verdict rule.

  • L1 Curated table: 243 domains across IFCN / Wikipedia RSP / primary-source / satirical / low-quality, plus subdomain → parent fallback (in-memory, O(1))
  • L2 Azure Tables SourceTierCache — fleet-wide, survives worker / pod restarts
  • L3 Batched Claude model-prior for anything still unknown; result writes back to L1 + L2
  • verdict Refuted / Supported needs ≥2 fact-checker / reputable-news on the central claim

curated lookup is < 1 ms; model-prior call (rare) is 30 s timeout

Stage 5 — Audit

mechanical

Pure-Python audit re-derives the verdict and validates every URL the renderer might surface is in the source-quality table. CoVe-style Claude audit is a planned follow-up.

  • checks verdict consistency · URL containment in justification AND load_bearing_evidence_snippet · one-bucket rule
  • on fail forces NotEnoughEvidence; NEI-shaped payload swapped in with reason="evidence retrieved but silent"

< 50 ms

Stage 6 — Freeze

data plumbing

Serializes the complete FrozenVerdict Pydantic tree to disk + an event row in Tables. Deep-immutable: nested models are frozen=True and every collection is a tuple.

  • disk data/freezes/<invocation_id>.json
  • tables MentionEvents row
  • pins Claude version · VLM version · source-list snapshot dates · git SHA

~100 ms

Stage 7 — Render

Claude reasoning

Claude renders one reply in the requested tone. The renderer sees ONLY RendererView (presentation_payload + tone_neutral_justification + state) — reasoning fields are unreachable by contract.

  • tones agreeable · neutral · agonistic — three independent prompts, no cross-references
  • invariance verbatim proper nouns · ≥1 cited URL (in verified state) · ≤256 X-weighted chars
  • timeout 30 s per call
  • refusal retry once with harm-reduction nudge; else raise → pipeline_error

~5–10 s · 1–2 retries on overflow

Stage 8 — X poster

data plumbing

Posts the rendered reply as a reply-to the original mention. The bot's /info URL is appended after the rendered text.

  • service X API v2 posts.create · OAuth 1.0a
  • cap 100 tweets / day (X Basic) · 500 soft per-tone (DERAD_MAX_MENTIONS_PER_DAY)
  • dry run DERAD_DRY_RUN=true logs but doesn't post

~1–2 s

Stage 9 — Outcome

data plumbing

Each accepted mention writes one event row. Engagement on each bot reply is polled hourly by a Container Apps cron job.

  • outcomes replied · empty_reply · pipeline_error · parent_fetch_failed · dry_run · pipeline_queue_timeout
  • variables likes · RTs · replies · QTs · target-user downstream sharing-quality (Mosleh)
  • tables MentionEvents · MentionDrops · EngagementSnapshots · BotReplyReplies

08 · optimizations

What's been tuned recently

Where the wall-clock and quota wins came from in the last build-week. Tail-latency mitigations & cost trims.

Concurrency wins

  • parallel Multi-image extraction fans out across 4 worker threads (was serial; ~30 s saved on 4-image tweets)
  • parallel HEAD-validation of search hits via 8-thread pool (was serial 6 s × top_k)
  • cap 25 concurrent pipelines · 10-min queue timeout

Skip the work that wouldn't matter

  • skip Annotation-stamped URLs short-circuit HEAD validation (server-vouched by tool runtime)
  • skip Step 0 seed search replaces the first Claude controller call (~3–5 s + tokens)
  • skip Reconcile bypassed when text empty AND no credible-tier image provenance (~5–10 s)
  • skip _short_circuit_no_checkable path skips Stage 4 + 4.5 entirely

Tail-latency caps

  • timeout Per-stage Claude budgets — 45 s extract · 30 s verify-controller · 90 s reconcile · 30 s render · 60 s VLM
  • timeout 15 s daemon-thread join on finalize-persist (Tables can't hold a semaphore slot)
  • timeout 60 s wall-clock budget per Stage 4 claim

Cache layers

  • L1 In-memory model-prior cache (per worker)
  • L2 Azure Tables SourceTierCache — one classification ever per domain fleet-wide
  • L3 Batched Claude classifier (only on cache miss)
  • lru_cache _git_sha · get_llm per (effort, tokens, timeout)

Cold-start trims

  • warmup Daemon thread builds the search-backend client at app startup so mention #1 doesn't pay credential-chain resolution + AIProjectClient init
  • lazy Participants store lazy-loads via write-through cache; no eager Tables scan per gunicorn worker

Prompt diet

  • trim Reconcile snippets capped at 240 chars; rationales at 120 chars (~20-30 % input token reduction)
  • trim tweet_context stripped of None/empty values before serialization
Net effect. Typical mention p50 dropped from 80–120 s → 50–90 s; tail p99 is bounded by the new per-stage timeouts instead of the single 120 s SDK default. Per-mention cost dropped from ~$0.15 → ~$0.12 (one fewer Claude call + tighter prompts).

09 · failure modes

How each stage degrades

Every stage is designed to fail toward NEI rather than toward a confident wrong reply.

Failure Detected by Behavior
Parent fetch returns nothing (deleted / private) Stage 1 Outcome parent_fetch_failed; bot drops the mention silently
Image too large or unreadable Stage 1.5 Image skipped, pipeline continues text-only
VLM extraction error Stage 1.5 This image's record marked unreadable, others continue
web_search 429 / tool error Stage 4 / 1.5 That question returns []; iterative controller decides whether to retry or stop
Claude 429 / per-stage timeout (30–90 s) Any LLM stage Narrow handlers in extract / verify / reconcile / multimodal catch (ValueError, TimeoutError) → stage-specific degradation. Unexpected exceptions propagate to pipeline_error.
Reconcile stance-count drift Stage 4.5 Pad/truncate to align with text-evidence count; emit reconcile_stance_drift metric tagged with delta size for observability
Tables write stalls (degraded backend) finalize log_mention_event runs on a daemon thread with 15 s join; semaphore slot releases even if persist never completes
Verdict label disagrees with structural rule Stage 5 audit Verdict forced to NotEnoughEvidence; NEI payload swapped in
Renderer cites a URL not in source-quality table Stage 7 runtime check One retry with the error as feedback; on second failure, empty_reply outcome
Renderer exceeds 280 X-weighted chars Stage 7 runtime check Same as above
Pipeline semaphore queue full for 10 minutes App-level throttle Outcome pipeline_queue_timeout; visibility in events table
X stream disconnect / 429 on reconnect Streamer thread Exponential backoff (capped at 30 min) until reconnect; mentions during disconnect are lost
Worker restart (deploy, scale, crash) App Service In-flight pipelines killed; atexit hook closes X stream so the next worker reconnects fast
