Fact-check pipeline

An invoked-bot architecture for fact-checking X posts. Nine stages move a mention from the X filtered stream through web-grounded verification, multimodal extraction, and tone-controlled rendering to a posted reply.

03 · zoom — stage 4

Inside the iterative verification loop

Stage 4 isn't a single search. A Claude controller decides — one step at a time — whether to keep searching, what to search for next, or whether the evidence is sufficient.

Caps

  • 5 max questions per claim
  • 3 hits per question (top-k)
  • 60 s wall-clock budget
  • 120 s per-call LLM timeout

Stop signals

  • stop ≥2 fact-checker or reputable-news hits with consistent stance
  • stop multiple searches, no relevant coverage → declare NEI
  • stop wall-clock or question cap hit

Continue signals

  • go primary source not yet checked (e.g. official press release)
  • go originating publisher unknown — look for retraction
  • go date/context detail would settle ambiguity
  • go author handle's authenticity not yet checked

04 · stages in detail

What each stage does, with what

One card per stage. Service/model is the actual deployment we hit at runtime.

Stage 1 — Ingestion

data plumbing

Service. X API v2 — filtered stream + posts.get_by_id.

What. The streamer (one persistent connection) matches @eddiexbot mentions. For each accepted mention the bot fetches the parent tweet (the tweet the mention replied to) along with author handle/bio/verified/created_at/followers, posted-at, expanded t.co URLs, referenced-tweet relations, and any attached photo URLs.

Out. TweetSnapshot + tweet_context dict.

~2 s · gated by X read-rate-limit (15 reads / 15 min)

Stage 1.5 — Multimodal extraction

Claude + Foundry+Bing

Conditional. Only fires when the parent has ≥1 photo.

Tier 1 (OCR). Claude VLM transcribes every readable string in the image.

Tier 2 (description). Same VLM call writes a literal, identifying description — instructed to name public figures (incl. musicians, actresses), landmarks, events, brands, canonical images with reasonable confidence.

Tier 3 (provenance approximation). Description fed back through Foundry+Bing to surface articles about the image's subject. True reverse-image-search has no Azure-native path post-Bing-API retirement.

Tier 4 (manipulation/AI-gen). Forced NEI per spec.

~7–13 s/image · cap 5 MB base64 (auto-resized via Pillow)

Stage 4 — Iterative verification

Claude + Foundry+Bing

Service. Claude Sonnet 4.6 (controller, low reasoning) + Azure AI Foundry agent (gpt-4.1-mini wrapping the bing_grounding tool).

What. Papelo-style loop. Each iteration: controller sees claim + history → decides continue (issues next question) or stop. The next question runs through Foundry+Bing, hits return as Evidence.

Caps. ≤5 questions · ≤3 hits/question · 60 s wall-clock · 120 s per LLM call.

~15–40 s · 1–3 questions typical · early-stops when ≥2 reliable-tier hits agree

Stage 4.5 — Reconciliation

Claude reasoning

Service. Claude Sonnet 4.6 with extended thinking (budget=4096).

What. One call folds together: Lens 1 (text-text), Lens 2 (image-text), Lens 3 (cross-modal). Sees claim, evidence, source-quality table, image evidence, and tweet_context. Emits atomic propositions (verified / refuted / disputed / unaddressed) with citations, a presentation_payload, a tone-neutral justification, and per-evidence stance labels.

Author signal. The model uses author handle + bio + verified + account-age to detect parody / fan / aggregator accounts and to date-stamp recency claims.

~30–50 s · the heaviest single LLM call in the pipeline

Stage 4.5.2 — Source reliability

structural rule

Four-step lookup. (1) curated IFCN signatory set; (2) Wikipedia perennial-sources style table (reputable-news / primary-source / satirical / low-quality); (3) subdomain → parent fallback (fact-check.usatoday.com → usatoday.com); (4) batched Claude model-prior classifier for the long tail, cached for the worker's lifetime.

Counts. 243 curated domains across five tiers. Anything else hits one Claude call per pipeline (batched).

Verdict rule. Refuted / Supported requires ≥2 sources at fact-checker or reputable-news tier on the central proposition. Conflicting / NEI otherwise.

Stage 5 — Verdict audit

mechanical

Service. Pure-Python checks (no LLM in current build; CoVe-style audit is a planned follow-up).

What. Re-derives the verdict label from consolidated_findings + source_quality_table via the structural rule and confirms it matches the declared label. Confirms every URL in the rendered justification is in the source-quality table. Forbids the central proposition appearing in more than one bucket.

On failure. Verdict forced to NotEnoughEvidence; NEI-shaped payload swaps in.

< 50 ms

Stage 6 — Freeze

data plumbing

Service. Local JSON + Azure Tables (events row).

What. Serializes the complete FrozenVerdict Pydantic object to disk under data/freezes/<invocation_id>.json. Deep-immutable: every nested model carries frozen=True and every collection is a tuple — list .append, nested attr writes, and list-item attr writes all raise.

Pins. Claude version, VLM version, search provider, source-reliability snapshot dates, pipeline git SHA.

~ 100 ms

Stage 7 — Tone renderer

Claude reasoning

Service. Claude Sonnet 4.6 — non-thinking, one of three system prompts per condition (agreeable / neutral / agonistic).

Invariance gate. Renderer receives only RendererView — presentation_payload + tone_neutral_justification. Reasoning fields are unreachable by contract. After rendering, a runtime check rejects any reply that introduces URLs outside primary_sources_to_cite or exceeds the X 280 char limit.

Hard constraints. Verbatim proper nouns (no "a satirical site" when the source said "World News Daily Report"). At least one cited URL. Retry-with-feedback on invariance failure.

~5–10 s · 1–2 retries on overflow

Stage 8 — X poster

data plumbing

Service. X API v2 — posts.create (OAuth 1.0a, single bot identity).

What. Posts the rendered reply as a reply-to the original mention from @eddiexbot. Refuses if X-weighted length > 280. Returns the new tweet ID for outcome tracking.

Daily cap. X Basic = 100 tweets / day total. Per-tone counter caps with DERAD_MAX_MENTIONS_PER_DAY=500 (soft).

Dry run. DERAD_DRY_RUN=true logs the would-be reply without posting.

~1–2 s

Stage 9 — Outcome tracker

data plumbing

Service. Azure Tables (MentionEvents, MentionDrops, EngagementSnapshots, BotReplyReplies) + a Container Apps cron job (poll-engagement).

What. Each accepted mention writes one event row with outcome (replied / empty_reply / pipeline_error / parent_fetch_failed / dry_run / pipeline_queue_timeout). Engagement snapshots polled hourly from the cron job.

Outcome variables. likes, retweets, replies, quote tweets, target-user subsequent sharing quality (Mosleh downstream-quality outcome), participant survey rows.

Per-mention cost (est.)

  • Foundry+Bing ~3–5 searches × ~$0.025 = ~$0.10
  • Claude 5–10 calls, ~50K tokens total ≈ ~$0.04
  • X API ~$0.013 per posted tweet at $200/100 daily cap
  • ≈ $0.15 / mention

Burst limits

  • 25 concurrent fact-check pipelines (semaphore cap)
  • 10 min queue timeout before drop (pipeline_queue_timeout)
  • 100/day X post cap (hard)
  • 500/day internal soft cap per tone

07 · failure modes

How each stage degrades

Every stage is designed to fail toward NEI rather than toward a confident wrong reply.

Failure Detected by Behavior
Parent fetch returns nothing (deleted / private) Stage 1 Outcome parent_fetch_failed; bot drops the mention silently
Image too large or unreadable Stage 1.5 Image skipped, pipeline continues text-only
VLM extraction error Stage 1.5 This image's record marked unreadable, others continue
Foundry 429 / Bing error Stage 4 / 1.5 That question returns []; iterative controller decides whether to retry or stop
Claude 429 / timeout (120 s) Any LLM stage Call returns error; pipeline records pipeline_error and skips reply
Verdict label disagrees with structural rule Stage 5 audit Verdict forced to NotEnoughEvidence; NEI payload swapped in
Renderer cites a URL not in source-quality table Stage 7 runtime check One retry with the error as feedback; on second failure, empty_reply outcome
Renderer exceeds 280 X-weighted chars Stage 7 runtime check Same as above
Pipeline semaphore queue full for 10 minutes App-level throttle Outcome pipeline_queue_timeout; visibility in events table
X stream disconnect / 429 on reconnect Streamer thread Exponential backoff (capped at 30 min) until reconnect; mentions during disconnect are lost
Worker restart (deploy, scale, crash) App Service In-flight pipelines killed; atexit hook closes X stream so the next worker reconnects fast
