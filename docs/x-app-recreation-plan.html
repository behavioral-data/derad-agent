Recreate the X App

The procedure for force-releasing a stuck filtered-stream slot by replacing the X App entirely. ~30–45 min hands-on time. Run this only after waiting ≥ 2 hours has not freed the slot.

when

When to run this

Don't run this prematurely. Recreating the X App is the documented force-clear for TooManyConnections, but it has a one-way cost: the App ID changes, all OAuth tokens issued by the old App become invalid, and any external materials pinned to the old Consumer Key need updating. Wait ≥ 2 hours of zero-retry quiet first.

Run this when

  • yes Streamer has been stuck on TooManyConnections for 2+ hours with the app stopped (zero retries hitting X).
  • yes The 30-min-backoff retries after that still 429 with connection_issue: TooManyConnections in the body.
  • yes You have @eddiexbot account access (login + any 2FA codes) to re-authorize.

Don't run this when

  • no The 429 body says anything OTHER than TooManyConnections — a standard rate-limit 429 fixes itself on its own.
  • no Less than 2 hours since the last retry.
  • no @eddiexbot 2FA / recovery codes are not handy.

Have ready before starting

  • CLI az login session active for subscription faac48db-165b-4928-a952-7d769267fe0b

az webapp stop --name azappspzdzrbtv3v4o --resource-group rg-derad-agent

phase 1

Snapshot current state for rollback

If anything goes sideways, we want to be able to restore the old secret values exactly. Snapshot to a local file (not git-tracked) before any mutations.

scripts/x-rollback-snapshot.sh (one-shot — outputs to a local file)
OUT=/tmp/x-app-rollback-$(date -u +%Y%m%dT%H%M%S).env
KV=azkvspzdzrbtv3v4o
{
  for name in x-api-key x-api-secret x-bearer-token x-access-token x-access-token-secret bot-user-id; do
    val=$(az keyvault secret show --vault-name "$KV" --name "$name" --query value -o tsv)
    echo "${name}=${val}"
  done
} > "$OUT"
chmod 600 "$OUT"
echo "Snapshot written to $OUT — keep it until cutover is verified."
The snapshot contains live secrets. Keep it on the dev machine only; delete it after Phase 7 verification succeeds. Do NOT commit it.

Also capture the OLD X App's settings from the portal (in case you need to mirror them on the new App):

  • note App name + description text
  • note OAuth 1.0a callback URL (likely oob)
  • note Permissions (Read / Write / DM)
  • note OAuth 2.0 client type (if any) + redirect URLs
  • note Project ID the App is attached to

phase 2

Create the new X App

  1. Go to developer.x.com/portal/projects-and-apps.
  2. Open the existing Project that the old App lives in. Don't create a new Project — the Project owns the access tier (Basic) and the new App must inherit it.
  3. Click Add App. Name it something distinct (e.g. derad-agent-v2) — easy to tell apart from the soon-to-be-deleted old App.
  4. Once created, copy the new Consumer Key and Consumer Secret shown on the success screen (they're shown ONCE; if you miss them, regenerate from the Keys & Tokens tab).
  5. Go to the new App's Settings:
    • permissions Set to Read and Write. (DM not needed — survey delivery is via tweet replies.)
    • app type Confidential client (web app).
    • callback Set OAuth 1.0a callback to oob (out-of-band PIN flow).
    • website Any plausible URL (e.g. https://uw.edu) — not validated.
  6. Back to Keys and Tokens → Bearer Token → Generate. Copy it.
Keep the OLD App alive for now. Don't delete it until Phase 8 — we want it as fallback during the rotation in case something on the new App is misconfigured.

phase 3

OAuth 1.0a 3-legged flow — re-authorize @eddiexbot

The new App needs the bot's user-context tokens to post replies. Run this script as the @eddiexbot user — it prints an authorize URL, you open it logged-in-as-@eddiexbot, click Authorize app, paste the PIN back.

scripts/x-oauth-pin.py (one-shot)
"""3-legged OAuth 1.0a PIN flow for X. Prints authorize URL, accepts PIN,
prints the resulting access_token + access_token_secret pair.

Usage:
    CONSUMER_KEY=... CONSUMER_SECRET=... python scripts/x-oauth-pin.py
"""
import os, sys, urllib.parse
import requests
from requests_oauthlib import OAuth1Session

CK = os.environ["CONSUMER_KEY"]
CS = os.environ["CONSUMER_SECRET"]

# Step 1 — request token
oauth = OAuth1Session(CK, client_secret=CS, callback_uri="oob")
r = oauth.fetch_request_token("https://api.x.com/oauth/request_token")
rt = r["oauth_token"]; rts = r["oauth_token_secret"]

# Step 2 — authorize
print(f"Open this URL while logged in as @eddiexbot:")
print(f"  https://api.x.com/oauth/authorize?oauth_token={rt}")
pin = input("PIN: ").strip()

# Step 3 — access token
oauth = OAuth1Session(CK, client_secret=CS, resource_owner_key=rt,
                     resource_owner_secret=rts, verifier=pin)
tok = oauth.fetch_access_token("https://api.x.com/oauth/access_token")
print()
print(f"X_ACCESS_TOKEN={tok['oauth_token']}")
print(f"X_ACCESS_TOKEN_SECRET={tok['oauth_token_secret']}")
print(f"BOT_USER_ID={tok.get('user_id')}")
print(f"BOT_HANDLE={tok.get('screen_name')}")
run it
pip install requests-oauthlib  # if not already installed
CONSUMER_KEY="<from-phase-2>" CONSUMER_SECRET="<from-phase-2>" python scripts/x-oauth-pin.py
  • verify The printed BOT_USER_ID matches the existing one in KV. If different, you authorized the wrong account — abort and redo.
  • verify BOT_HANDLE is eddiexbot.
  • note If the PIN flow errors with "callback_uri mismatch", go back to Phase 2 and confirm OAuth callback is oob.

phase 4

Test new credentials locally before touching KV

Three quick tests to confirm the new App works before we commit by updating Key Vault.

test 1 — bearer token can connect to filtered stream
BEARER="<new-bearer-from-phase-2>"
curl -sS -m 5 -D /tmp/hdrs.txt -o /tmp/body.txt -w 'HTTP %{http_code}\n' \
  -H "Authorization: Bearer $BEARER" \
  'https://api.x.com/2/tweets/search/stream?tweet.fields=author_id'
# Expected: HTTP 200 (curl will hold the stream for 5s then disconnect — fine)
# A 429 here means the new App somehow inherited the old slot — extremely unlikely
test 2 — OAuth1 user-context can post (DRY: just lookup, don't post)
CK="..." CS="..." AT="<new-access-token>" ATS="<new-access-token-secret>" python - <<'PY'
import os
from requests_oauthlib import OAuth1Session
s = OAuth1Session(os.environ["CK"], client_secret=os.environ["CS"],
                  resource_owner_key=os.environ["AT"], resource_owner_secret=os.environ["ATS"])
r = s.get("https://api.x.com/2/users/me")
print(r.status_code, r.json())
# Expected: 200 with the @eddiexbot user data
PY
test 3 — bearer-token rate-limit budget is fresh (50/15min remaining)
BEARER="..."
curl -sS -D - -o /dev/null -m 5 \
  -H "Authorization: Bearer $BEARER" \
  'https://api.x.com/2/tweets/search/stream/rules' | grep -i 'x-rate-limit'
# Expected: x-rate-limit-remaining: 449 or 450 (fresh budget for the rules endpoint)
If any test fails: stop here. Don't update KV. Re-check Phase 2 + 3, regenerate the bearer token if needed, redo the OAuth flow. The new App's slot is still pristine — no harm done.

phase 5

Update Azure Key Vault

Set new values for the 5 X-credential secrets. bot-user-id stays unchanged.

update all 5 secrets
KV=azkvspzdzrbtv3v4o
az keyvault secret set --vault-name $KV --name x-api-key             --value "<NEW_CONSUMER_KEY>"             --output none
az keyvault secret set --vault-name $KV --name x-api-secret          --value "<NEW_CONSUMER_SECRET>"          --output none
az keyvault secret set --vault-name $KV --name x-bearer-token        --value "<NEW_BEARER_TOKEN>"            --output none
az keyvault secret set --vault-name $KV --name x-access-token        --value "<NEW_ACCESS_TOKEN>"            --output none
az keyvault secret set --vault-name $KV --name x-access-token-secret --value "<NEW_ACCESS_TOKEN_SECRET>"     --output none
echo "All 5 X secrets updated"

App Service env vars reference KV via @Microsoft.KeyVault(SecretUri=...) — the URI doesn't pin a version, so the latest secret value is resolved on next refresh (≤ 30 s on app restart). Container Apps job env vars use secretRef, resolved on next job invocation.

phase 6

Restart App Service with new credentials

start the app — it pulls new KV values on container start
az webapp start --name azappspzdzrbtv3v4o --resource-group rg-derad-agent

The new container picks up:

  • New X_BEARER_TOKEN → streamer connects to filtered stream with new App identity → fresh slot from X's edge perspective.
  • New X_API_KEY / X_API_SECRET / X_ACCESS_TOKEN / X_ACCESS_TOKEN_SECRET → get_x_client() builds the OAuth1 client with the new identity.
  • Existing 300 s streamer pre-connect delay still applies (defense in depth — but the slot is already fresh, so first connect should succeed at the 300 s mark).

phase 7

Verify the cutover

Health + stream state

  • curl https://azappspzdzrbtv3v4o.azurewebsites.net/healthz → 200
  • curl /api/activity → metrics.stream_connected: true after the 300 s pre-delay
  • logs Look for "Filtered stream connected" in App Service logs

End-to-end mention test

  • manual From a test account: tweet @eddiexbot what's the context here? on any factual claim
  • verify Bot replies within ~60 s
  • verify Reply tweet ends with the /info/... short link
  • verify Click the link → new info page renders with action pill + outcome + sources
Halt criterion: if stream_connected is still false 10 minutes after the 300 s pre-delay (so ~15 min after start), check logs. If you see a TooManyConnections 429 from the NEW bearer token, abort and proceed to rollback — something's wrong with the new App config or X is having a degradation unrelated to the slot issue.

phase 8

Cleanup

  • delete The OLD X App in the developer portal. This is the final force-release step on X's side. (We waited for the new App to be verified working before deleting — old App's slot, even if still held, no longer matters.)
  • delete The local rollback snapshot file (/tmp/x-app-rollback-*.env) — it contains stale but still-sensitive secrets.
  • delete Any test tweets the bot posted during Phase 7.
  • update agent/llm/.env locally with new values (gitignored — manual edit).
  • note No code commit needed — secrets are infra-level only.

if anything breaks

Rollback (Phases 5–6 only)

Rollback only works if the OLD App hasn't been deleted. Phase 8 is intentionally last so this window stays open. If you've already deleted the old App, rollback isn't possible — you have to fix-forward on the new App.
restore old KV secrets from the Phase 1 snapshot
SNAP=/tmp/x-app-rollback-<timestamp>.env  # from Phase 1
KV=azkvspzdzrbtv3v4o
while IFS='=' read -r name value; do
  [ -z "$name" ] && continue
  # only restore the X-* secrets; bot-user-id we left alone anyway
  case "$name" in
    x-*) az keyvault secret set --vault-name $KV --name "$name" --value "$value" --output none ;;
  esac
done < "$SNAP"
az webapp restart --name azappspzdzrbtv3v4o --resource-group rg-derad-agent
echo "Rolled back to OLD App credentials."

risks

Known risks & mitigations

RiskMitigation
App deletion is irreversible from X's side. Phase 8 is LAST. Verify new App works in Phase 7 first. Keep old App alive for the rotation; delete only after the new one is proven.
OAuth re-auth invalidates existing user tokens. The old App's tokens still work until we update KV in Phase 5. Phase 4 tests the new tokens BEFORE updating KV, so KV is only updated when we know new tokens work.
Lose the rollback snapshot before verification. Phase 1 outputs to /tmp/x-app-rollback-<ts>.env with mode 0600. Phase 8 deletes it only after verification.
New App's Project might not inherit the Basic plan tier. Phase 2 step 2 explicitly says "create in the existing Project" — the Project owns the tier. Don't create a new Project.
Tests in Phase 7 succeed but the bot replies look wrong (no sources / wrong action). This is a code issue, not a rotation issue. Roll back the X creds is irrelevant; investigate the pipeline / freeze records. Rotation itself doesn't change any code paths.
Per-user-day cap (DERAD_USER_DAILY_CAP=20) or pipeline semaphore (DERAD_PIPELINE_MAX_CONCURRENT=5) doesn't reset on App switch — Tables state persists. Expected. The bot's "memory" lives in Tables, not in the X App. If a participant has already hit their daily cap, the cap stays in effect.
Container Apps engagement-poll cron job picks up new creds on next invocation. Next cron tick is typically < 60 min. If a manual sooner run is needed: az containerapp job start --name azcjob-engspzdzrbtv3v4o --resource-group rg-derad-agent.
Existing /info/<token> tokens still resolve. Yes — info tokens live in Azure Tables, not in X. They reference parent tweet IDs (which keep working) and reply tweet IDs (which keep working). No action needed.
Participants need to re-onboard / re-mention the bot. No — they mention by handle (@eddiexbot), and the handle is unchanged. Stream rules get re-synced automatically on first connect by _sync_rules.
