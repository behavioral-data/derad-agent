derad-agent

Given a polarizing social media claim, derad-agent plans search queries, retrieves semantically similar tweets from a pre-built Community Notes index, and asks an LLM to compose a grounded reply. The LLM reads note text as private background knowledge — it never cites sources or reveals the retrieval process.

LLM Azure OpenAI chat or embeddings call compute Pure Python / NumPy, no network artifact File on disk

Phase 2 · Every query

Runtime pipeline

Entry points: derad-ask --statement "…" CLI or retrieve_statement_landscape() Python API.

compute
0 · Load index
Read all three artifacts into a NotesIndex dataclass. L2-normalise the embeddings matrix once — from this point, dot products equal cosine similarities.
Cost: one-time disk read per process. Memory: ~4 bytes × N × D for the matrix.
LLM
1 · Query planner
The claim is passed to the LLM with a prompt asking it to decompose it into 3–4 focused search queries. Falls back to the raw statement if JSON parsing fails.
reasoning_effort=low · max_tokens=2000 · output: {"queries": ["q1", …]}
LLM · embed
2a · Embed each query
Each query string is embedded with the same Azure OpenAI model used at build time. Result is L2-normalised.
One API call per query. Output: float32 [D].
compute
2b · Cosine retrieval
embeddings @ query_vec over all N tweets gives a similarity score per row. Partition the top k×4, sort, then walk the list applying the exclude list and similarity_min threshold until k results are collected.
k_per_query=25 (default) · similarity_min=0.0 (default) · output: [(tweet_id, sim), …]
compute
3 · Dedup across queries
Results from all queries are merged into a single dict keeping the best similarity score per tweet. Sorted descending by score.
Output: unique List[(tweet_id, best_sim)]
compute
4 · Note slice
For each retrieved tweet, take the first notes_per_tweet notes from the cache (already helpful-only and recency-sorted). Attach the tweet's similarity score. Flatten to a single list.
notes_per_tweet=10 (default) · no filtering or sorting needed — the index guarantees both.
LLM
5 · Reply composer
Note summaries are cleaned and packaged as an evidence JSON payload. The LLM receives a style preamble + the claim + the evidence and writes a direct social-media reply as if it simply knows these facts. It is explicitly instructed not to mention notes, sources, datasets, or any retrieval process.

If the response JSON fails to parse, a repair LLM call converts the raw text back to valid JSON.
reasoning_effort=low · max_tokens=1400 · preamble: neutral | bridging | agonistic
output: {"response": "…", "reasons": [{reason, note_id, tweet_id, evidence_links}]}
output
Output
response — the social media reply. Shown to the user.
reasons — citations keyed to note_ids. Internal tracing only, never shown to the user.
sources — deduplicated URLs extracted from reasons. Printed below the reply in the CLI.
Full JSON saved to results/ask_runs/{timestamp}.json.

Runtime · tone

Response styles

All three styles run identical retrieval. Only the LLM system preamble changes.

StylePreamble intentWhen to use
neutral default Knowledgeable bystander replying with facts. No framing. General use. Safe default.
bridging Acknowledge what's understandable about the concern, then add context. Calm, inclusive language. When the poster is likely sincere and open to new information.
agonistic Direct challenge. Name the error clearly. Respectful but no softening. When the claim is clearly wrong and directness is warranted.

Reference

Design decisions

Key choices and the reasoning behind them.

Proxy embeddings — embedding note summaries instead of tweet text
We have no access to tweet text (Twitter API v2 is not part of this setup). Instead, we embed the concatenated note summaries for each tweet as a proxy for the tweet's content. This works because note authors describe what the tweet claims — the summaries are topically dense and written specifically to characterise the tweet's assertion. The limitation is that retrieval quality is bounded by how well the summaries represent the original claim space.
Pre-filter and pre-sort at build time, not runtime
Only CURRENTLY_RATED_HELPFUL notes enter the index. Non-helpful notes are discarded at build time, not at query time. Similarly, each tweet's notes are sorted by recency before being written to the cache. This means the runtime select_recent_helpful_notes is a pure slice with no conditional logic — faster and simpler. The trade-off is that the index must be rebuilt when note statuses change (Community Notes publishes regular data snapshots, so this fits naturally into a periodic rebuild cadence).
Fixed N notes per tweet for embedding (default 15)
Embedding models have token limits (~8k tokens for Azure OpenAI models). A note summary is typically 50–150 words, so uncapped concatenation would hit the limit for popular tweets with many helpful notes and get silently truncated. Using a fixed cap (most recent N) ensures consistent token budget and embedding quality across all tweets. 15 was chosen as a balance between coverage and staying well within limits. The most recent notes are preferred because they are most likely to reflect the current consensus — older notes may be superseded. N is configurable via --notes-for-embedding.
Notes as private LLM context — the LLM does not cite them
The evidence payload is passed to the LLM as background knowledge the person being replied to cannot see. The LLM prompt explicitly forbids mentioning notes, evidence, sources, datasets, or any retrieval process. The goal is a reply that reads as a knowledgeable person responding naturally — not as a fact-checker citing a database. The reasons array in the output is internal bookkeeping (note_ids, tweet_ids, URLs) used for tracing and evaluation; it is never surfaced to the end user.
Multi-query retrieval with best-score dedup
A single embedding of the claim may not capture all its facets (e.g. "Vaccines cause autism" involves both vaccine safety and autism causation). The LLM planner generates 3–4 queries that decompose the claim. Each query retrieves independently, and results are merged by keeping the best similarity score per tweet. This increases recall without duplicating tweets in the final list.
Response styles are preamble-only
All three styles (neutral, bridging, agonistic) run identical retrieval and receive identical evidence. Only the system preamble injected before the claim changes. This keeps retrieval logic simple and makes style a pure presentation choice rather than a data choice. A/B testing different styles on the same claim is trivially cheap — reuse the retrieved notes, swap the preamble.
