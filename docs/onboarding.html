DeRad Agent — Collaborator Onboarding

Everything you need to go from zero to full dev+deploy access.

What this system does

The DeRad Agent monitors mentions of a single X (Twitter) bot account (@eddiexbot) via the X Filtered Stream API. When a user mentions the bot while replying to a post containing a claim, the agent fetches the parent post, searches Community Notes for relevant evidence, picks a reply tone (the registered participant's assigned tone, or a random tone for unregistered users), generates the reply with an LLM, and posts it. All events are stored in Azure Table Storage.

streaming Ingest

X Filtered Stream → mention dispatch → dedup/rate-limit guard

LLM Pipeline

Fetch parent tweet → embed query → retrieve notes → generate reply (GPT / Grok)

storage Output

Post reply on X → write MentionEvent row to Azure Tables → log to App Insights

Step 1 — Get Azure access

Ask Advait to run this once with your UW email: 
bash scripts/grant-access.sh you@uw.edu
This grants Contributor on the resource group, Key Vault Secrets User, Storage Table Data Contributor, and AcrPush — everything you need.

Once granted, verify with:

az login
az account set --subscription faac48db-165b-4928-a952-7d769267fe0b
az webapp show --name azapplikxqqfjcgk72 --resource-group rg-derad-agent --query name -o tsv
# should print: azapplikxqqfjcgk72

Step 2 — Clone and install

git clone <repo-url>
cd derad-agent
pip install -e .

Python 3.9+ required. A conda or venv environment is recommended.

Step 3 — Bootstrap your .env

All secrets live in Azure Key Vault. The setup script pulls them all and writes agent/llm/.env for you — no manual credential sharing needed. 

bash scripts/setup-env.sh
Never commit .env — it is in .gitignore. Re-run setup-env.sh after key rotations.

The script sets DERAD_INGEST_MODE=off so your local app never opens a second stream connection and disrupts production. Do not change this.

Step 4 — Run locally

flask --app agent.app.app run --port 5001

The dashboard is at http://localhost:5001/dashboard. With ingest off, the stream panel will show disconnected — that's expected. You can still use all CLI tools and test the pipeline manually.

Step 5 — Deploy to production

Only one person deploys at a time. Coordinate on Slack before pushing a new image — a restart drops the stream connection for ~60 seconds.

Build and push a new image

# From the repo root — builds in Azure, no local Docker needed
az acr build \
  --registry azacrlikxqqfjcgk72 \
  --image derad-agent:latest \
  --file Dockerfile \
  . 2>&1 | tail -5

Deploy the new image

# Get the new digest
DIGEST=$(az acr repository show \
  --name azacrlikxqqfjcgk72 \
  --image "derad-agent:latest" \
  --query digest -o tsv)

# Pin App Service to it (triggers restart)
az webapp config container set \
  --name azapplikxqqfjcgk72 \
  --resource-group rg-derad-agent \
  --container-image-name "azacrlikxqqfjcgk72.azurecr.io/derad-agent@$DIGEST"

Verify the deployment

# Watch the live log stream for "Filtered stream connected"
curl -s "https://azapplikxqqfjcgk72.azurewebsites.net/stream/logs" | grep "data:" | grep -E "stream|rule|error"

Critical: X Filtered Stream

The X API allows only one active filtered stream connection per bearer token. If a local app starts the streamer (ingest mode = streaming), it will steal the connection from production, causing 429 errors and bot downtime.

Always keep DERAD_INGEST_MODE=off in your local .env. The setup-env.sh script sets this automatically.
