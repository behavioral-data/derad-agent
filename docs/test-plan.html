Study workflow

End-to-End Test Plan

Three phases — in-memory smoke test, live dry-run, and full live — covering participant registration through day-3 engagement collection.

Phase A

Smoke test

Pure in-memory, no API calls. Runs in ~2 seconds. Verifies all logic before touching live infrastructure.

python scripts/smoke_test.py
Phase B

Live dry-run

Real Azure Tables + Azure OpenAI. DERAD_DRY_RUN=true so no X posts. Verifies pipeline output quality and event logging.

Phase C + D

Full live + Day-3

Real X mention → bot reply. 3 days later: poll engagement and collect bystander replies. Run derad-daily-summary.

Phase A · No external APIs

Smoke test

Verifies the full data-flow logic end-to-end in memory. Must pass before proceeding to live phases.

Run it
python scripts/smoke_test.py
What it tests (7 phases)
  • done Phase 1 — InMemoryParticipantsStore.register() and list_all()
  • done Phase 2 — App startup reads participant store; participant appears in _PARTICIPANTS_BY_ID and _ALLOWED_IDS
  • done Phase 3 — Unregistered author dropped with reason=unregistered; registered author accepted
  • done Phase 4 — process_mention() sets study_code (4-char, valid alphabet), study_day=6 (enrolled May 15, received May 20), participant_id
  • done Phase 5 — _poll_one() writes EngagementSnapshot with correct metrics and mention_id back-link
  • done Phase 6 — _collect_one() writes BotReplyReply with correct text, username, like count
  • done Phase 7 — derad-daily-summary prints per-participant table with study codes and URLs
Expected output: all assertions print ✓ and the script exits 0. A ✗ FAIL line means a regression — do not proceed to live phases until green.

Phase B · Real infra, no X posts

Live dry-run

Register a test participant in the real Participants table, start the app with DERAD_DRY_RUN=true, trigger a dispatch via the provided test script, and verify the MentionEvent row in Azure Tables.

Step B1 — Register a test participant
derad-register-participant \
  --author-id YOUR_X_USER_ID \
  --username yourxhandle \
  --tone neutral \
  --enrolled 2026-05-20 \
  --notes "smoke test participant"

Writes a row to the Participants Azure Table. Requires DERAD_PARTICIPANTS_BACKEND=tables and DERAD_TABLES_ENDPOINT.

Step B2 — Start the app in dry-run mode
DERAD_DRY_RUN=true \
DERAD_INGEST_MODE=off \
gunicorn derad_agent.app.app:app \
  --bind 0.0.0.0:5000 \
  --workers 1 \
  --timeout 120

INGEST_MODE=off prevents the X stream from starting. We inject test mentions directly.

Step B3 — Inject a test mention via Python
python - <<'EOF'
import os, sys
os.environ["DERAD_EVENTS_BACKEND"] = "tables"
os.environ["DERAD_PARTICIPANTS_BACKEND"] = "tables"
os.environ["DERAD_DRY_RUN"] = "true"
os.environ["DERAD_INGEST_MODE"] = "off"
os.environ.setdefault("SERVER_NAME", "localhost:5000")

from datetime import datetime, timezone
from derad_agent.app import app as app_module
from derad_agent.app import dedup as dedup_module

tweet = {
    "id_str": "TEST_MENTION_001",
    "in_reply_to_status_id_str": "SOME_REAL_TWEET_ID",
    "user": {"id_str": "YOUR_X_USER_ID", "screen_name": "yourxhandle"},
    "text": "@neutralbot This climate legislation is very effective.",
}
app_module._dispatch_tweet("neutral", tweet, datetime.now(timezone.utc))
print("Dispatch sent — check logs and Azure Tables for MentionEvent.")
EOF
Step B4 — Verify in Azure Tables

Check the MentionEvents table for a row with:

FieldExpected value
mention_idTEST_MENTION_001
outcomedry_run
participant_idYOUR_X_USER_ID
study_day1 (enrolled today)
study_codenull (dry_run exits before reply is posted)
reply_textNon-empty LLM response
study_code is null in dry-run. The code is derived from the reply tweet ID, which is only assigned after X accepts the post. In dry-run mode the bot exits before posting, so study_code is never set. This is expected.

Phase C · Everything real

Full live test

Send a real X mention from the test participant's account. Verify the bot replies, then verify the MentionEvent in Azure Tables has all study fields set.

Step C1 — Start the app in streaming mode
gunicorn derad_agent.app.app:app \
  --bind 0.0.0.0:5000 \
  --workers 1 \
  --timeout 300

Requires DERAD_INGEST_MODE=streaming (now the default) and X_BEARER_TOKEN.

Step C2 — Send a test mention

From the participant's X account, post a reply to any public tweet that mentions the Eddie bot:

@eddiexbot This new climate legislation seems really effective at reducing emissions.

The tweet must be a reply (not a quote tweet or standalone mention). The bot will only respond when in_reply_to_status_id_str is set.

Step C3 — Verify MentionEvent fields
FieldExpected
outcomereplied
reply_idNon-null tweet ID
study_code4-char code, e.g. BZCH
participant_idParticipant's X user ID
study_dayDays since enrollment + 1
reply_textNon-empty, cites community notes
pipeline_msUnder 30,000 (30 s)
Step C4 — Check healthz and logs
curl http://localhost:5000/healthz
# → {"ok": true, "index_loaded": true}

Phase D · 3 days after C

Day-3 collection

Steps D1 and D2 run automatically each day at 12:00 UTC via the engagementCronJob Container Apps Job. The commands below are for manual ad-hoc runs (e.g. to backfill a missed day before it ages out of the window).

Step D1 — Poll engagement metrics (automated)
derad-poll-engagement

Reads all reply_ids from MentionEvents whose reply_posted_utc is in the [3d, 4d) window. Fetches public metrics from X and writes an EngagementSnapshot row per reply. The cron job invokes this daily.

Step D2 — Collect bystander replies (automated)
derad-collect-replies

Searches the conversation thread for replies to each bot post in the 3-day window. Writes BotReplyReply rows for NLP analysis. Uses the same [3d, 4d) window so it runs alongside derad-poll-engagement in the daily cron.

Step D3 — Daily summary for survey DMs
derad-daily-summary --date $(date -d yesterday +%Y-%m-%d)

Prints a table grouped by participant showing study codes and bot-reply URLs. Use this to compose the daily DM survey links.

=== Daily DM Summary for 2026-05-20 ===

@studyparticipant (id=123456, bot=@eddiexbot, tone=neutral, day=6)
------------------------------------------------------------
  [BZCH] https://x.com/eddiexbot/status/1234567890
          (original: https://x.com/i/web/status/9876543210)

Total: 1 replies across 1 participant(s)
Step D4 — Send survey DM
python -m derad_agent.app.survey \
  --participant-id PARTICIPANT_X_USER_ID \
  --form-url "https://forms.gle/your-form-url" \
  --tone neutral

Sends a Google Form link by X Direct Message from the assigned bot account.

Reference

Go / no-go checklist

Before running live phases
  • Smoke test passes: python scripts/smoke_test.py exits 0
  • Full test suite passes: python -m pytest tests/ -q → 90 passed
  • DERAD_TABLES_ENDPOINT is set and reachable
  • Notes index is built and at INDEX_ROOT
  • Test participant registered: derad-register-participant --author-id ... --tone neutral
  • App restarted after participant registration (allow-list is loaded at startup)
  • /healthz returns {"ok": true, "index_loaded": true}
Day-3 timing window
derad-poll-engagement and derad-collect-replies only pick up replies whose reply_posted_utc is in the [72h, 96h) window. The engagementCronJob Container Apps Job fires daily at 12:00 UTC — if it skips a day (job failure, image pull issue), replies posted in that 24-hour band are silently lost. Check the Container Apps run history if any day's snapshot counts look low.
