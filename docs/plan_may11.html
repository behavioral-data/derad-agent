01 · Engineering — this week

LLM relevance filter for community notes

The retrieval step currently passes all top-K notes to the composer. Add an LLM filter that drops notes unrelated to the statement before composition, reducing noise and improving reply quality.

Why: Cosine similarity over embeddings retrieves notes that are topically adjacent but not always relevant to the specific claim. A fast LLM pass (Haiku-class) filtering by note ID is cheap and keeps the composer context clean.

Step 1 — Prompt + factory

start here
  • Write RELEVANCE_FILTER_TEMPLATE prompt in llm/prompts.py. Input: statement + JSON list of {note_id, summary} pairs. Output: JSON array of IDs to keep.
  • Add get_relevance_filter_prompt(statement, notes_json) factory next to the template.

Step 2 — Filter step

next
  • Create runtime/steps/relevance_filter.py with step_filter_notes_by_relevance(statement, notes). Call LLM with the prompt, parse returned JSON list of IDs, filter notes list. Fallback to original list on parse error.
  • Export new step from runtime/steps/__init__.py.

Step 3 — Wire into agent + tests

next
  • Add filter_notes: bool = True param to run_landscape_agent in runtime/landscape_agent.py. Log how many notes survived the filter.
  • Unit tests in tests/test_relevance_filter.py: (a) LLM returns subset of IDs, (b) malformed JSON → fallback, (c) empty input → no LLM call.
  • Smoke test with indexes_5pct/ data — print notes before/after filter on a claim from conditions.md.
